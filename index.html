Executive Summary

Angular v22 is a stability and platform-expansion release.

Angular v22 focuses on making modern Angular production-ready. The release graduates three important capabilities to stable: Signal Forms, Angular Aria, and asynchronous reactivity APIs such as resource and httpResource.

Stable

Signal-first Angular

Forms and async data loading now align more closely with Angular Signals.

Stable

Accessible UI primitives

Angular Aria helps teams build accessible custom components without locking into one visual design.

Developer Experience

Better templates

Template comments, spread syntax, improved @switch, and arrow functions reduce boilerplate.

Why Angular v22 is important

  • It makes Signal Forms production-ready for teams building complex enterprise forms.
  • It makes resource and httpResource production-ready for declarative async data.
  • It strengthens Angular's AI development story using MCP, Agent Skills, and WebMCP.
  • It improves performance defaults with OnPush for new applications.
  • It prepares the ecosystem for future builder and TSGo-focused improvements.

Main Angular v22 Features

Production Ready

1. Signal Forms

Signal Forms is Angular's new forms API that combines strong typing, reactive forms ideas, template-driven simplicity, and signal-based reactivity.

Before

Forms were usually handled with FormGroup, FormControl, validators, and observables.

After

Forms can be modeled with a signal-backed model and schema-based validation.

Real-world use case

Payment forms, profile forms, onboarding flows, enterprise workflow forms, and admin configuration screens.

import { signal } from '@angular/core';
import { form, required } from '@angular/forms/signals';

export class PaymentComponent {
  readonly paymentModel = signal({
    paymentType: '',
    amount: 0
  });

  readonly f = form(this.paymentModel, schema => {
    required(schema.paymentType, {
      message: 'Payment type is required'
    });
  });
}
<form>
  <label for="payment-type">Payment Type</label>

  <select id="payment-type" [formField]="f.paymentType">
    <option value="">Select payment method</option>
    <option value="credit">Credit Card</option>
    <option value="paypal">Payment Service</option>
  </select>

  @if (f.paymentType().invalid() && f.paymentType().touched()) {
    @for (error of f.paymentType().errors(); track error.kind) {
      <p class="error">{{ error.message }}</p>
    }
  }

  <button type="submit" [disabled]="f().invalid()">
    Submit Payment
  </button>
</form>

Best practices

  • Keep form model structure close to your business DTO.
  • Use schema validation for reusable validation rules.
  • Prefer Signal Forms for new complex forms in Angular v22.

Common mistakes

  • Mixing too much manual DOM state with Signal Forms.
  • Putting large validation logic directly inside templates.
  • Migrating every old form at once instead of gradually.
Production Ready

2. Angular Aria

Angular Aria provides accessible, headless UI patterns. You bring your own styles and business logic; Angular Aria handles accessibility behavior like keyboard interaction, ARIA roles, and focus patterns.

Before

Teams had to manually implement ARIA behavior for custom dropdowns, menus, trees, and tabs.

After

Teams can build custom design-system components using tested accessibility primitives.

Real-world use case

Design systems, banking dashboards, enterprise admin tools, custom menus, custom trees, and accessible component libraries.

<!-- Example concept: custom accessible menu -->
<button aria-haspopup="menu" aria-expanded="false">
  Actions
</button>

<div role="menu">
  <button role="menuitem">Edit</button>
  <button role="menuitem">Duplicate</button>
  <button role="menuitem">Delete</button>
</div>
  • Use Angular Aria when building custom UI components instead of reinventing accessibility behavior.
  • Test keyboard navigation and screen-reader behavior, not only visual design.
Production Ready

3. Asynchronous Reactivity: resource and httpResource

resource and httpResource allow Angular applications to handle async state using a signal-friendly mental model.

Before

Developers used observables, manual loading flags, error flags, and subscriptions.

After

Async data can be represented as a resource with value, loading, error, and reactive parameters.

import { resource, signal, computed } from '@angular/core';

const selectedCity = signal('Pune');

const weatherResource = resource({
  params: () => ({ city: selectedCity() }),
  loader: ({ params }) => fetchWeatherForecast(params.city),
});

const currentTemperature = computed(() => {
  if (weatherResource.hasValue()) {
    return `${weatherResource.value().temperature}°C`;
  }

  return 'Loading weather...';
});
import { signal } from '@angular/core';
import { httpResource } from '@angular/common/http';

export class WeatherComponent {
  selectedCity = signal('Pune');

  weather = httpResource<{ temperature: number; condition: string }>(() => {
    return `/api/forecast/${this.selectedCity()}`;
  });

  changeCity(city: string) {
    this.selectedCity.set(city);
  }
}

Best practices

  • Use httpResource for simple HTTP-driven UI state.
  • Use resource when the async source is not only HTTP.
  • Keep complex stream transformations in RxJS when they are truly stream-based.
Performance Default

4. OnPush is Default for New Applications

Angular v22 aligns new applications with performance-first rendering by making OnPush the default. Components no longer need to specify OnPush explicitly in new applications.

Before

Developers had to write changeDetection: ChangeDetectionStrategy.OnPush manually.

After

New apps use OnPush by default, reducing unnecessary change detection work.

// Angular v22 new application style
@Component({
  selector: 'app-weather',
  template: `<section>Loading Weather...</section>`
})
export class WeatherComponent {
  // OnPush behavior by default in new applications
}

The old default strategy has also been renamed to ChangeDetectionStrategy.Eager to make its behavior clearer. 

@Component({
  selector: 'app-legacy-widget',
  template: `<section>Legacy Widget</section>`,
  changeDetection: ChangeDetectionStrategy.Eager
})
export class LegacyWidgetComponent {}
New API

5. @Service Decorator

Angular v22 introduces @Service() as a clearer way to define global singleton services. It improves readability compared with @Injectable({ providedIn: 'root' }) for common use cases.

Before

@Injectable({ providedIn: 'root' }) was used for most application-wide services.

After

@Service() communicates intent more directly for singleton services.

import { Service } from '@angular/core';

@Service()
export class BasicDataStore {
  private data: string[] = [];

  addData(item: string): void {
    this.data.push(item);
  }

  getData(): string[] {
    return [...this.data];
  }
}
Performance API

6. injectAsync

injectAsync enables asynchronous dependency injection. This means large services can be loaded only when needed, enabling service-level code splitting.

Before

Routes and components could be lazy-loaded, but services were commonly loaded upfront.

After

Heavy services can be imported on demand and even prefetched when idle.

import { Component, injectAsync } from '@angular/core';

@Component({
  selector: 'app-report',
  template: `<button (click)="export()">Export</button>`,
})
export class ReportComponent {
  private exporter = injectAsync(() => import('./report-exporter'));

  async export() {
    const exporter = await this.exporter();
    exporter.export();
  }
}
export class ReportComponent {
  private exporter = injectAsync(() => import('./report-exporter'), {
    prefetch: onIdle,
  });
}

Router Updates

Angular v22 improves navigation and route cleanup.

Platform Navigation API Integration

The router can integrate with the browser's native Navigation API. This helps with navigation interception, native scroll behavior, loading indicators, and accessibility announcements. 

bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(appRoutes, withExperimentalPlatformNavigation())
  ]
});

Route Cleanup Controls

withExperimentalAutoCleanupInjectors helps destroy injectors connected to inactive routes. This can prevent memory leaks in large route-heavy applications. 

import {
  provideRouter,
  withExperimentalAutoCleanupInjectors,
} from '@angular/router';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(
      routes,
      withExperimentalAutoCleanupInjectors(),
    )
  ]
};

Template Improvements

Angular v22 makes templates more expressive and easier to maintain.

Comments inside elements

You can document bindings and attributes directly inside an element.

Spread syntax

Object, array, and function-call spread syntax can be used in templates.

Better @switch

Multiple cases can share output, and exhaustive checking can use @default never.

Arrow functions

Small inline arrow functions are now valid in templates.

Host directive de-duplication

Angular automatically de-duplicates matching host directives.

Better maintainability

These changes reduce boilerplate and help humans and AI agents understand templates.

Template comments

<div
  // valid comment
  /* another valid comment */
  attr1="value1"
  /*
    comment spanning multiple lines
  */
  attr2="value2">
</div>

Spread syntax in templates

<section>
  <div [class]="{
    ...standardCupStyles,
    'cardboard-sleeve': isHotDrink
  }">
  </div>

  <app-bakery-cart
    [pastryOrder]="[...dailyPastryBasics, 'croissant', 'muffin']" />

  <p>
    Total Cost: {{ calculateBill(...baseItemPrices, salesTax) }}
  </p>
</section>

Improved @switch

@switch (orderStatus) {
  @case ('Pending')
  @case ('Processing') {
    <p class="badge-blue">In progress</p>
  }

  @case ('Shipped') {
    <p class="badge-green">On its way!</p>
  }

  @default never;
}

Arrow functions in templates

<p>Stock: {{ item().stock }}</p>

<button
  (click)="item.update(p => ({ ...p, stock: p.stock - 1 }))">
  Decrease Stock
</button>

Production Best Practices

Recommended practices for Angular v22 projects.

Use Signals intentionally

Use signals for local and derived state. Keep RxJS for complex stream orchestration.

Adopt Signal Forms gradually

Start with new forms, then migrate old forms where it gives real maintainability value.

Use Angular Aria for custom UI

Accessibility should be designed into your component system, not added at the end.

Prefer OnPush mental model

Write components assuming explicit, predictable state updates.

Use injectAsync for heavy services

Great for export, PDF, charting, analytics, and rarely used enterprise tools.

Monitor deprecations

Plan migration away from deprecated Webpack builder paths.

Common Mistakes

What teams should avoid while adopting Angular v22.

  • Using new APIs everywhere without understanding production impact.
  • Replacing RxJS completely even where streams are the better abstraction.
  • Ignoring accessibility because Angular Aria is available.
  • Putting long-running arrow functions directly inside templates.
  • Using injectAsync for small services that do not need lazy loading.
  • Not reviewing Webpack builder deprecations before upgrading build pipelines.
  • Assuming OnPush default means performance is automatic without good state design.

Migration and Breaking-change Notes

Important upgrade considerations for Angular v22.

Area What changed Recommended action
Webpack builders Webpack support and related build tooling are deprecated. Move toward the modern Angular application builder and watch TSGo-related updates.
Change detection OnPush is default for new apps; previous default is renamed Eager. Review assumptions in legacy components and state mutation patterns.
Forms Signal Forms are production-ready. Use for new complex forms; migrate old forms gradually.
Async data resource and httpResource are production-ready. Use for simple declarative async state; keep RxJS where needed.
Router Experimental navigation and cleanup APIs are available. Use carefully and test memory behavior in large apps.

Interview Questions

Useful questions for Angular v22 preparation.

Q1. What are the three major production-ready features in Angular v22?

Signal Forms, Angular Aria, and asynchronous reactivity APIs such as resource and httpResource.

Q2. Why is Signal Forms important?

It gives Angular a modern, signal-based forms API that is strongly typed, declarative, reactive, and suitable for complex enterprise forms.

Q3. When should you use httpResource instead of HttpClient with observables?

Use httpResource for simple UI-driven data fetching where parameters depend on signals. Use RxJS when you need advanced stream composition.

Q4. What is injectAsync?

It is an async dependency injection API that allows large services to be lazy-loaded only when needed.

Q5. What changed in Angular change detection?

OnPush is now the default for new applications, and the previous default strategy is renamed ChangeDetectionStrategy.Eager.

Q6. What is WebMCP?

WebMCP is an experimental way for web apps to expose structured browser tools that AI agents can use directly.

Final Summary

Angular v22 is about production readiness, AI readiness, and performance-first defaults.

Angular v22 shows a clear direction: build applications with signals, ship accessible user interfaces, simplify async data handling, support AI-assisted workflows, improve templates, and move toward faster, cleaner build infrastructure.

  • Use Signal Forms for modern form-heavy applications.
  • Use Angular Aria for accessible custom UI components.
  • Use resource and httpResource for signal-friendly async data.
  • Use injectAsync for large lazy-loaded services.
